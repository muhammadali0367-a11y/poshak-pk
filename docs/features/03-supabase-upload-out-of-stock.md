# Feature: Supabase Upload & Out-of-Stock Detection

## 1. What It Does

Takes the enriched product CSV and syncs it to the Supabase PostgreSQL database. This isn't a simple insert — it performs three critical operations in a single run:

1. **Upsert products:** Insert new products and update existing ones (price changes, name changes, etc.)
2. **Mark out-of-stock products:** Any product that exists in the database but is absent from the latest CSV gets its `in_stock` flag set to `false`.
3. **Generate and sync embeddings:** For each new or updated product, creates an OpenAI embedding and upserts it to Pinecone for vector search.

This is the bridge between the Python scraping pipeline and the live Next.js website.

---

## 2. How It Works Technically

### Upsert Logic

The script reads `output/all_brands_poshak.csv` and processes products in batches. For each product:

1. Uses the stable UUID (generated by `convert.py` from the product URL) as the primary key.
2. Calls Supabase's `upsert()` — if the ID exists, it updates all fields; if not, it inserts a new row.
3. This means price changes, name changes, and category corrections are automatically applied every night.

### Out-of-Stock Detection (`mark_out_of_stock`)

After all products are upserted, the script runs a per-brand comparison:

1. Collects all product IDs from the CSV, grouped by brand.
2. Queries Supabase for all existing product IDs for that brand where `in_stock = true`.
3. Any ID present in Supabase but absent from the CSV → `UPDATE SET in_stock = false`.

This works correctly because of stable UUIDs — the same product URL always maps to the same ID. If Khaadi removes a product from their site, the scraper won't find it, the CSV won't contain its ID, and `mark_out_of_stock` sets `in_stock = false`.

**Why per-brand:** If the scraper failed for one brand (e.g., Sapphire returned 0 products due to blocking), only Sapphire's products get checked. Other brands are unaffected.

### Embedding Generation

For each product that's new or whose `embedding_synced` flag is `false`:

1. Concatenates: `name + tags + category + brand` into a single string.
2. Sends to OpenAI's `text-embedding-3-small` model to get a 1536-dimensional vector.
3. Upserts the vector to Pinecone with the product's UUID as the vector ID.
4. Sets `embedding_synced = true` in Supabase.

This happens inline during the upload — no separate batch job needed.

| File | Role |
|---|---|
| `upload_to_supabase.py` | Main upload script. Reads CSV, upserts to Supabase, marks out-of-stock, generates embeddings, logs pipeline run. |

| Supabase `pipeline_runs` table | Receives logging data about the upload |
| Pinecone `poshak-products` index | Receives product embeddings |

## 4. APIs Used

| API | Purpose | Cost |
|---|---|---|
| Supabase REST API (via Python client) | Upsert products, query existing IDs, update in_stock flags | Free (within free tier) |
| OpenAI Embeddings (`text-embedding-3-small`) | Generate 1536-dim vectors for each product | ~$0.000002 per product |
| Pinecone Upsert API | Store product vectors for search | Free (within 100k vector limit) |

## 5. Edge Cases

- **Scraper fails for one brand:** If `scraper.py` returns 0 products for Khaadi but succeeds for all other brands, `mark_out_of_stock` will mark ALL Khaadi products as out of stock. This is technically correct behavior (the latest scrape found 0 Khaadi products) but is usually caused by a scraper bug, not actual stock removal. Currently there's no safeguard against this.
- **UUID migration:** If a product's URL changes (e.g., brand restructures their site), it gets a new UUID. The old UUID stays in the database marked out of stock, and a new row is created. This can lead to brief duplicates during the transition period.
- **Embedding API failure:** If OpenAI's embedding API is down during upload, products still get upserted to Supabase but `embedding_synced` stays `false`. They'll be picked up in the next run's embedding pass. However, they won't appear in vector search results until embedded.
- **Pinecone free tier limit:** At 100k vectors with ~31k products currently, there's headroom. But if the catalog grows to 80k+ products, vectors from deleted products (still in Pinecone) could push over the limit. There's no orphan vector cleanup.
- **Concurrent pipeline runs:** If a new pipeline run starts while the previous one is still uploading, both could be writing to Supabase simultaneously. Railway's cron schedule prevents this in normal operation, but manual triggers could cause it.
- **CSV encoding issues:** Product names with special characters (Urdu, accented characters) could cause CSV parsing issues. UTF-8 encoding is used throughout, but malformed data from the scraper could still slip through.

## 6. How to Improve

- **Brand-level scrape validation:** Before running `mark_out_of_stock`, check if a brand's product count dropped by more than 50% compared to the previous run. If so, flag it as a potential scraper failure rather than actual stock removal.
- **Orphan vector cleanup in Pinecone:** After marking products out of stock, also delete their corresponding vectors from Pinecone. Currently, out-of-stock products still have vectors and can appear in search results (though they're filtered at the Supabase query level).
- **Batch embedding calls:** Instead of one embedding API call per product, batch multiple product strings into a single call (OpenAI supports up to 2048 inputs per request). Would dramatically reduce API call count and speed up the embedding phase.
- **Diff-based upload:** Instead of upserting every product every night, compute a diff between the CSV and the database. Only upsert products that actually changed. Would reduce Supabase write operations by 90%+ on stable nights.
- **Rollback capability:** Store the previous night's product snapshot so that if a pipeline run produces bad data, you can quickly restore the previous state without waiting for another full scrape cycle.
- **Embedding versioning:** If the embedding model changes (e.g., upgrading from `text-embedding-3-small` to a newer model), all existing vectors become mismatched with new query embeddings. Add a version field so you can trigger a full re-embed when the model changes.
